# Contributing to Gastty

Thanks for taking the time. This file is the short version of "how this project is wired so your PR doesn't surprise anyone."

## Quick start

```sh
brew install zig@0.15 xcodegen
git clone --recursive git@github.com:hampusaberg/Gastty.git
cd Gastty
./scripts/build-libghostty.sh      # ~5 min first run, cached after
xcodegen                            # regenerates Gastty.xcodeproj
open Gastty.xcodeproj               # ⌘R to build + run
```

If you pulled new commits and something doesn't compile, the usual fix is `xcodegen` (someone added a file) and/or `git submodule update --init --recursive` (someone bumped libghostty).

## Layout in 30 seconds

- **`project.yml`** is the source of truth — Xcode's `.xcodeproj` is gitignored and regenerated by `xcodegen`. Don't hand-edit the project; edit `project.yml`.
- **`vendor/ghostty/`** is a git submodule pinned to a specific upstream commit. Treat it as read-only; if you need a libghostty change, bump the submodule SHA in its own PR.
- **`Sources/TerminalApp/`** is all the Swift code. The README has a per-file map.
- **`Resources/AppIcon.icns`** is regenerated from `scripts/generate-icon.swift` whenever `AppDelegate.makeAppIcon()` changes. Re-run the script and commit the new `.icns`.

## Architecture cheatsheet

- **AppKit, not SwiftUI**, for window/menu/responder-chain reasons. SwiftUI is only the `@main` Scene anchor.
- **Menu actions use the responder chain.** Define `@objc func someAction(_:)` on the class that should handle it (often `AppDelegate` or `TerminalWindowController`), leave the menu item's `target = nil`, and AppKit routes it. The Connections sidebar's gear button is a good example.
- **`ConnectionStore`, `SettingsStore`, `GhosttyRuntime`** are singletons; they post `Notification`s on mutation and observers refresh.
- **Persistence** lives in `Persistence/AppPersistence.swift`. If you add restorable state, bump the schema with backwards-compat decoding (see `ConnectionStore.PersistedV2` for the v1→v2 pattern).
- **libghostty integration** flows through `GhosttyRuntime` (singleton app + action dispatcher) and `SurfaceHostView` (the NSView hosting one `ghostty_surface_t`). Keyboard input is in `SurfaceInput.swift` — note the PUA function-key carve-out for arrows.

## Code style

- **SwiftLint** runs in CI. Local: `brew install swiftlint && swiftlint Sources/`. Warnings don't fail the build; errors do.
- **Comments earn their keep.** Explain *why*, not *what*. Most existing files have a short doc comment at the top of each type explaining its role — match that tone.
- **No `// MARK: -` spam.** Group with one `MARK: - Section` per actual section; don't sprinkle them above every function.
- **Avoid `force unwraps` on optionals you can't prove are non-nil.** `guard let` is fine, especially in AppKit callbacks where `nil` is a legitimate "view tore down" signal.

## Adding a feature — typical flow

1. Read the file you're modifying end to end first. Existing patterns matter more than your preferred ones.
2. Make the smallest change that solves the problem. Bigger refactors land in their own PR.
3. If you add a Swift file, run `xcodegen` so it appears in the project, then build to verify.
4. Keep diffs focused. Mixing a feature + a refactor + a typo fix in one PR makes review painful.

## Pull requests

- Branch off `main`. Push to your fork (or a feature branch in this repo if you're a collaborator).
- Open the PR against `main`. CI runs automatically — needs to be green to merge.
- PR title and body matter: title is a one-liner that goes into the merge commit. Body should explain the *why* and call out anything tricky a reviewer would otherwise miss. Match the style of recent commits (`git log --oneline -20` is your reference).
- The build matrix is currently just macOS. Test locally on whatever macOS version you have; CI catches the rest.

## Reporting bugs / suggesting features

Open an issue. The issue forms (bug / feature request) will prompt you for the bits we need — please fill them in rather than free-form so we don't have to ask follow-ups.
